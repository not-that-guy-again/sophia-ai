# ADR-028: Outbound notification service

**Date:** 2026-03
**Status:** Accepted

## Context

Sophia can receive external events via webhooks (ADR-025) but has no way to proactively communicate with customers. When a shipment is delayed, a return is processed, or a payment fails, the event router can update memory and trigger the pipeline, but the resulting action has no outbound channel.

Real customer service requires proactive communication. A human CS agent would email or message a customer when their shipment is delayed. Without outbound notifications, Sophia's proactive pipeline triggers (ADR-025) produce decisions that go nowhere — the agent decides "notify the customer about the delay" but has no mechanism to do so.

Alternatives considered:

- **Direct SMTP / Twilio integration** (rejected: couples Sophia to specific delivery providers, requires managing credentials and delivery infrastructure, duplicates what MCP servers already provide).
- **Expose a "pending notifications" API and let the customer's system poll** (considered for future: avoids Sophia owning delivery, but adds latency and integration burden on the customer). This is effectively what the `WebhookNotificationService` provides.
- **Let the pipeline output a notification action that the tool executes** (partially adopted: the pipeline decides to notify, but the delivery mechanism is a separate service, not embedded in tool logic. Tools stay thin wrappers per ADR-022).
- **No outbound capability — leave it to the customer** (rejected: makes Sophia's proactive features useless in practice, since the event router triggers a pipeline that can only talk to itself).

## Decision

A `NotificationService` interface is introduced following the established provider abstraction pattern (ADR-009, ADR-015, ADR-022):

```python
class NotificationService(ABC):
    async def send_notification(
        self, recipient: NotificationRecipient, message: NotificationMessage,
    ) -> NotificationResult: ...

    async def get_channels(self) -> list[str]: ...
```

Framework-owned dataclasses define the notification contract: `NotificationRecipient` (customer_id, email, phone, channel preference), `NotificationMessage` (subject, body, source event), `NotificationResult` (success, channel, message_id, failure_reason).

### Implementations

Three implementations ship:

1. **LogNotificationService** — Default for development. Logs the notification to the audit system without sending it. Zero external dependencies.

2. **WebhookNotificationService** — POSTs the notification payload to a configurable URL. The customer's own system handles actual delivery. This is the lightest production integration — Sophia decides what to say, the customer's infrastructure decides how to deliver it.

3. **MCPNotificationService** — Uses MCP servers for delivery (ADR-026 pattern). Gmail MCP for email, Twilio MCP for SMS, etc. Same adapter pattern as backend services: Sophia's `NotificationMessage` is mapped to MCP tool arguments, and the MCP result is mapped back to `NotificationResult`.

### Safety Gate

Outbound notifications pass through a `NotificationGate` before delivery. The gate enforces:

1. **Per-customer rate limit** — No more than N notifications per customer per day (configurable in hat config, default 3). Prevents notification storms from rapid-fire webhook events.

2. **Quiet hours** — No notifications during configurable hours (default 9pm–8am). If a notification is generated during quiet hours, it is queued and sent when the window opens. Timezone is resolved from the customer's address if available, otherwise defaults to the hat's configured timezone.

3. **Content check** — The notification body is scanned for common issues before sending: compensation promises that exceed agent authority, personally identifiable information that shouldn't be in notifications, and language that contradicts the hat's tone. This is a lightweight heuristic check, not a full pipeline run.

Blocked or queued notifications are logged to the audit system with the reason.

The gate is intentionally separate from the consequence engine. Running the full evaluation pipeline for every notification would add 3–5 seconds of latency and multiple LLM calls. The gate's checks are deterministic and sub-millisecond. The consequence engine already evaluated the action that produced the notification — the gate only checks delivery-level concerns.

### Notification Message Generation

The message body is generated by the `ResponseGenerator` using the hat's system prompt and tone. This ensures proactive messages sound consistent with the agent's conversational voice, not like system-generated alerts.

The `EventRouter` passes the `SophiaEvent` data to the response generator with a notification-specific prompt template:

```
Generate a brief, customer-friendly notification about this event.
Event: {event_type} for {entity_type} {entity_id}
Details: {event_data}
Keep it under 3 sentences. Be informative, not alarming.
```

### Configuration

The notification service is configured per-Hat in `hat.json`:

```json
{
  "notifications": {
    "provider": "webhook",
    "config": {
      "url_env": "NOTIFICATION_WEBHOOK_URL",
      "auth_token_env": "NOTIFICATION_WEBHOOK_TOKEN"
    },
    "limits": {
      "max_per_customer_per_day": 3,
      "quiet_hours_start": 21,
      "quiet_hours_end": 8,
      "timezone": "America/New_York"
    }
  }
}
```

The `notifications` block is optional. When absent, no outbound notifications are sent and `memory_update_and_notify` falls back to `memory_update` silently.

### Integration with Event Router

The `EventRouter` gains a `notification_service` dependency alongside `memory` and `agent_loop`. For `memory_update_and_notify` actions:
1. Memory is updated (same as `memory_update`)
2. Customer contact info is resolved from the memory system
3. A notification message is generated via the response generator
4. The notification gate checks rate limits and quiet hours
5. If the gate approves, the notification service delivers the message
6. The result (sent, blocked, queued) is logged to audit

## Consequences

**Positive:**
- Sophia can proactively reach customers when external events warrant communication, making webhook-triggered pipeline runs actionable.
- The provider pattern means the notification delivery mechanism is swappable without changing the event router or pipeline.
- The safety gate prevents notification abuse (storms, off-hours messages) without the overhead of full pipeline evaluation.
- The `WebhookNotificationService` option lets customers control their own delivery infrastructure while Sophia controls the content and timing.
- The `LogNotificationService` default preserves zero-dependency development — developers see what notifications would be sent without configuring email or SMS.

**Negative:**
- Adds another service interface and provider pattern, increasing the number of abstractions a contributor must understand.
- The notification gate's content check is heuristic-based and may miss issues that the full evaluation pipeline would catch. This is an intentional trade-off for latency.
- Quiet-hours queueing requires a persistence mechanism for queued notifications. The initial implementation uses an in-memory queue that is lost on restart. Production deployments with quiet-hours requirements need the notification queue backed by the database.
- MCP-based notification delivery inherits the MCP ecosystem's maturity limitations (ADR-026). Email MCP servers may not support all formatting options, SMS servers may have provider-specific quirks.
- The notification message is LLM-generated, adding one LLM call per outbound notification. For high-volume event streams, this cost could be significant. Template-based notifications (no LLM call) should be considered as a future optimization.
